# You.md — Gap vs Vision (2026-06-11)

One row per promised value prop. Evidence and finding numbers (#N) reference PRODUCT-AUDIT.md; tasks reference FEATURE-ROADMAP.md.

| Promise | Current reality | Gap | Fix |
|---|---|---|---|
| "Agents always meet the same version of you" | Four different versions: web agent (bundle+30 memories), CLI chat (bundle, zero memories), hosted MCP (5 tools, no memory/private despite its own tool description), GitHub repo (last manual push) — `useYouAgent.ts:186-261`, `chat.ts:2255-2266`, `http.ts:2788-2854`, `GithubRepoSection.tsx:42-66` | Core promise unmet at every entry point except web (#3, #7, #19) | One `assembleAgentContext()` for all surfaces + hosted MCP parity + auto-push to repo (Tasks 2.1, 2.2, 2.7) |
| "Stop re-explaining yourself to every agent" | Memories cannot be searched (category+newest-100 only, `memories.ts:28-46`), durable preferences silently decay at 90/180 days (`memories.ts:331-374`), categories differ across 4 surfaces so filters miss data | You re-explain anything older than 90 days or beyond 200 memories (#6, #16, #17) | Search index + `q` param, decay exemption for durable categories, one taxonomy (Tasks 2.3, 2.4, 2.10) |
| "One elegant layer" | Six dot-conventions (`~/.youmd`, cwd `.youmd/`, `.you/`, `.youmd-project`, `youstacks/`, `stacks/`), four project-context roots with two schemas, cwd bundle silently shadows the home brain (`config.ts:61-69`) | Which "you" loads depends on cwd and code path (#10, #14, #22) | Canonical stack layout, one project-context engine, documented precedence + status visibility (Tasks 2.5, 3.8, 3.15) |
| Scoped API keys for agents (least-privilege access) | Scopes stored and plan-gated but never enforced — `authenticateRequest` discards them (`http.ts:385-429`); a free `read:public` key writes bundles and reads private context | Critical security/trust gap; the contract is decorative (#1) | `requireScope()` on every route + MCP tool (Task 1.1) |
| "76 endpoints + 24 MCP tools" | Counts are honestly machine-generated, but the hosted MCP agents actually discover via `.well-known/mcp.json` has 5 tools; the 76 includes a retired Clerk webhook + 3 internal-token admin routes | Zero-install agents get ~20% of the promised tool surface (#7, #27) | Hosted parity; until then split the claim hosted-5/local-24 and exclude retired/internal (Tasks 2.2, 3.10) |
| Installable, portable YouStacks | No `stack install`, no registry, no stacks table; CLI cannot discover stacks in the user's own canonical repo (3 incompatible layouts); routing endpoint echoes caller-supplied capabilities | "Installable" means manual file copying; the moat layer is CLI-locked prose (#10, #11) | `youmd stack install` from repo mirror + canonical layout + public registry (Task 2.5) |
| Trustworthy agent capability contract | `GET /api/v1/stacks/capabilities` advertises per-stack endpoints, scopes, and MCP tools that do not exist — agents following it hit 404s (`youstack-routing.ts:57-95`) | A protocol that publishes false contracts (#12) | Rewrite contract to real surfaces tagged by transport; CI curl test (Task 2.6) |
| Draft → publish control over your public identity | Every save instantly syncs to the public profile (`me.ts:494-536`); rollback returns success but changes nothing publicly (`bundles.ts:258-273`) | No actual draft boundary; rollback is a public no-op (#2) | Pick save==publish explicitly; rollback patches profile in same mutation (Task 1.2) |
| "Preferred local setup": `youmd mcp --install claude --auto` | Writes `mcpServers` into `~/.claude/settings.json`, which Claude Code never reads (`mcp.ts:82-122`) — promoted across llms.txt, docs, installer | Flagship onboarding is a verified no-op (#8) | Target `~/.claude.json` / `claude mcp add` + post-install verify (Task 1.3) |
| `youmd sync` keeps local and cloud in sync | One-shot sync pulls (unconditional overwrite) then pushes the remote's content back (`sync.ts:84-93`, `pull.ts:72-98`) | Silently destroys local unpushed edits (#5) | Dirty-check + push-first ordering + 3-way merge via existing `base.json` (Task 0.2) |
| One consistent runtime everywhere | npm `youmd@latest` = 0.6.21, repo/docs = 0.6.23, curl installer clones unpinned GitHub HEAD — three concurrent versions; npx-launched MCP always stale | The product promising "the same version of you" ships three versions of itself (#9, #36) | Publish 0.6.23, pin installs, version-skew CI gate (Tasks 0.3, 3.11) |
| Cross-agent activity monitoring ("who read/wrote what") | Substrate is excellent (before/after hashes, agent detection) but generated MCP configs never set `YOUMD_AGENT_NAME`, so every host logs as "Claude Code" (`server.ts:505`) | Attribution is fiction across hosts (#35) | Per-host env in generated configs + `clientInfo.name` fallback (Task 1.3/3.10) |
| Private vs public boundary you can trust | Owner-auth sweep is solid, but security tokens gating private data use `Math.random()` (#4), API keys sit world-readable on disk (#30), and `youmd pull` writes private notes into committable cwd `.youmd/` with no gitignore (#37) | Private data protected by guessable tokens and loose file hygiene | Secure token helper, 0600 perms, auto-gitignore (Tasks 0.1, 0.4, 0.6) |
| Self-improving agent-first platform (reference intelligence) | Real harness exists (`scripts/reference-intelligence.mjs`) but is human-triggered only; no schedule, no You Agent eval suite | A self-improvement *script*, not a loop (#28) | Weekly cron PR + golden Q&A evals in CI (Task 3.12) |
| Agent-native API ergonomics | Strong where shipped (ETag/304, SSE, conflict 409s) but no pagination (directory truncates at 500), no idempotency keys, two error shapes with no codes, rate limits on 7 routes only, no webhooks/subscriptions — agents must poll | Protocol plumbing below the bar the vision sets (#15, #23-#26) | Cursors, idempotency, error codes, limits+headers, outbound webhooks (Tasks 2.9, 3.1-3.4) |

**Bottom line:** the substrate (versioned bundles, conflict semantics, auth sweep, docs generation, activity log) is genuinely strong and ahead of peers. What's missing is coherence: one context assembly, one stack layout, one runtime version, and contracts (scopes, drafts, capabilities, install paths) that do what they say. Milestones 0-2 of the roadmap close the gap between pitch and product.
